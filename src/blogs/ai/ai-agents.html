本文梳理 AI Agent 的核心范式（ReAct / Plan-and-Execute / Reflexion）， 并以 LangGraph 为例展示生产级 Agent 的工程实践。

什么是 AI Agent

AI Agent 是能够感知环境、基于大模型推理、并通过工具调用影响环境的系统。 与传统 RAG 不同，Agent 强调自主规划、迭代决策的能力。

一个最小可用的 Agent 包含三要素：

  • 推理核心：LLM 负责思考、规划、决策
  • 工具集：可调用的函数（搜索、代码执行、API 等）
  • 记忆：短期（对话）+ 长期（向量库 / KV）

ReAct 范式

ReAct（Reason + Act）是最经典的 Agent 范式。Agent 在每一步中 先推理（Reason）下一步该做什么，再行动（Act）调用工具， 然后根据工具返回观察（Observation）结果继续推理，直到得到最终答案。

执行流程

Thought 1: 用户想知道天气，我需要调用天气 API
Action 1: get_weather(city="Hangzhou")
Observation 1: { "temp": 22, "condition": "sunny" }
Thought 2: 已经获得天气，可以回答用户了
Final Answer: 杭州今天晴天，气温 22 度。

伪代码示例

async function react_agent(query, llm, tools):
  history = [{ role: "user", content: query }]
  for step in range(MAX_STEPS):
    thought = await llm(history, tools=tools)
    if thought.action == "FINAL":
      return thought.answer
    observation = await tools[thought.action](**thought.args)
    history.append({ role: "tool", content: observation })
  return "超出最大步数"

Plan-and-Execute 范式

ReAct 适合单步推理，但面对复杂多步任务时容易"短视"。 Plan-and-Execute 先让 LLM 生成完整计划，再逐步执行：

  1. Planner：根据用户输入生成步骤列表
  2. Executor：依次执行每一步，可调用 ReAct 风格的子 Agent
  3. Replanner：根据执行结果动态调整计划

LangChain 的 PlanAndExecute agent 和 LangGraph 的 state machine 都是该范式的工程实现。

Reflexion：让 Agent 学会反思

Reflexion 引入"反思记忆"：Agent 完成一次尝试后，会用自然语言总结 哪里做错了，下次再遇到类似任务时调用这些反思。论文报告在 HumanEval 上提升约 10%。

生产实践要点

成本控制

Agent 调用次数多，token 消耗是普通对话的 5-20 倍。常用优化：

  • 小模型路由：简单任务用 GPT-4o-mini，复杂任务才升级到 GPT-4o
  • 工具结果截断：超过 4k token 的搜索结果先压缩
  • 缓存：相同 query 复用结果

可靠性

Agent 的失败模式包括：无限循环、参数幻觉、超时。 工程上必须实现：

  • 最大步数限制（MAX_STEPS）
  • 工具调用的 schema 严格校验
  • Trace 记录（LangSmith / Phoenix）
  • 关键操作的 human-in-the-loop 确认

评测

Agent 评测比普通 LLM 更复杂，因为同一 query 多次跑结果可能不同。 建议从三个维度评估：

  • 成功率：是否在规定步数内给出正确答案
  • 平均步数：反映效率
  • 成本：单次任务的 token 消耗

小结

AI Agent 的核心不在于"调用工具"这一表层动作，而在于持续的推理 - 行动 - 观察闭环。 生产级 Agent 的难点是可靠性、成本、可观测性，而非 prompt 编写本身。 建议从 ReAct 起步，遇到复杂任务再演进到 Plan-and-Execute + Reflexion。
