单次调用能解决的不上 Workflow，Workflow 能解决的不上 Agent。 每一级复杂度都带来更高的延迟、成本和调试难度。 选择依据是任务是否真正需要那个级别的自主性。

"Start with the simplest solution possible, and only increase complexity when needed." — Anthropic

一、方法论的演进：从 Prompt Engineering 到 Harness Engineering

LLM 应用工程经历了三个阶段，每个阶段的出现都由新的任务需求催生。

1.1 Prompt Engineering（2022—2023）

Prompt Engineering 关注优化单次调用 LLM 的文本指令：Role 设定、few-shot 示例、Chain-of-Thought 引导。

局限：随着 Agent 进入多轮交互、工具调用、长会话，「一段精心写好的 prompt」不再够用—— 模型在不同步骤需要的信息是动态变化的。

1.2 Context Engineering（2025 上半年）

2025 年 6 月，Tobi Lütke（Shopify CEO）和 Andrej Karpathy 几乎同时提出 "Context Engineering"， 随后被 Simon Willison、LangChain（Harrison Chase）、Anthropic 共同认可。 Anthropic 在 Effective Context Engineering for AI Agents（2025-09-29）中给出权威定义：

"the set of strategies for curating and maintaining the optimal set of tokens (information) during LLM inference"

Context Engineering 关注整个上下文窗口的内容：系统提示、工具定义、检索结果、历史消息、状态、压缩策略—— 确保模型在每一步都看到「刚好够用的信息」。

局限：上下文工程仍然停留在单个上下文窗口内。而当 Agent 需要跨会话、跨上下文窗口持续执行任务时， 需要在复杂的任务环节中稳定地运行，只对上下文内容进行优化是不够的。

1.3 Harness Engineering（2025 末—2026）

2025 年 11 月，Anthropic 在 Effective Harnesses for Long-Running Agents 中首次将 Harness 概念体系化。

2026 年 2 月 5 日，Mitchell Hashimoto（HashiCorp 创始人）在 My AI Adoption Journey 中给出了 Harness Engineering 的实践哲学：

"anytime you find an agent makes a mistake, you take the time to engineer a solution such that the agent never makes that mistake again."

6 天后，OpenAI 的 Ryan Lopopolo 发表 Harness Engineering: Leveraging Codex in an Agent-First World， 记录用 Codex 写出约 100 万行代码、零人类编写的实验，从工程实践层面正式确立 Harness Engineering 的地位。

Harness Engineering 关注模型外围的整套基础设施：工具集、AGENTS.md、结构性测试、验证脚本、子 Agent 编排、运行环境、权限边界、跨会话状态。 目的是「让 Agent 长期、自主、安全地运行」。

Harness Engineering 仍然包括 Prompt 和 Context 的设计，只是进一步在更高层面思考问题：
怎么让模型可以一直自主运行？怎么让出错的概率下降或者让 Agent 能够有效纠正错误？以及怎么让用户更好地能「驾驭」模型？

二、当前 Agent 的两大类型

2.1 编排型 Agent（Workflow Agents）

代表项目：LangChain / LangGraph、Dify、n8n、Coze

特征：开发者用图（DAG / StateGraph）搭好执行路径，模型在指定节点做单步推理。 控制流由人为设定，模型不参与路由判断。适合用在专业性的应用中，人为搭建稳定可靠的工作流。

2.2 自主型 Agent（Autonomous Agents）

代表项目：Claude Code、OpenAI Codex、OpenClaw、Hermes Agent

特征：模型在一个 loop 中自主完成「plan → 调用工具 → 观察结果 → 反思 → 继续或停止」的循环。 控制流由模型在运行时根据任务进展动态决定。适用于探索性、开放性、灵活性强的任务场景中。

2.3 两者与 Harness 的关系

个人观点：编排型 Agent 关键在于人为编排工作流和优化每个节点的执行效果——这是 Harness 的一部分实现方式，但不是 Harness 的全貌。 Harness Engineering 更多地是针对自主型 Agent 要如何设计其工具调用、运行环境、记忆加载、任务编排、安全权限机制等的工程性实现方法，以提升用户对 Agent 的「驾驭」。

  • 编排型 Agent 根据人为搭建的工作流做决策——把约束和流程编排人为写死在流程图里。
  • 自主型 Agent 给模型决策权——需要通过工具设计、上下文管理、权限边界、可观测性等来引导模型行为。

三、Agent 构建基石：增强型 LLM（Augmented LLM）

在讨论 Workflow 和 Agent 之前，先理解它们共同的基础构件。增强型 LLM 是在基础模型之上叠加三种能力的系统：

  • 工具调用（Tool Use）：让模型能够主动调用外部函数（搜索、计算、文件操作）
  • 记忆（Memory）：会话内的短期记忆 + 跨会话的长期记忆（向量库 / 文件 / KV）
  • 检索（Retrieval）：从外部知识库中拉取与当前任务相关的信息

这三种能力是 Workflow 与 Agent 的共同底座——无论上层走「编排」还是「自主」路线，都依赖它们之中的至少一种来突破「纯文本对话」的局限。

四、架构详解

1️⃣ 基础架构：LLM 单次调用

最简单的 LLM 应用就是一次 prompt → response 的单次调用。 适合单步推理任务：摘要、翻译、改写、问答、分类等。 优势：最简单、可预测、易调试；劣势：没有外部信息源、没有迭代修正。 

user ──prompt──> LLM ──response──> user

2️⃣ 工作流（Workflows）架构

把 LLM 调用嵌入到预先设计好的执行流程中，控制流由开发者定义，模型只负责在指定节点做单步推理。 典型范式包括：Prompt Chaining、Parallelization、Orchestrator-Workers、Evaluator-Optimizer。 

            ┌─→ LLM ──┐
input ──→   ├─→ LLM ──┼──→ aggregator ──→ output
            └─→ LLM ──┘
   (Parallelization 示意)

适用场景：流程固定、可拆解、可验证的任务，例如多阶段文档处理、并行多角度分析等。

3️⃣ 自主 Agent（Agents）架构

模型在一个 loop 中自主完成「plan → 调用工具 → 观察结果 → 反思 → 继续或停止」的循环。 工具调用、状态推进、终止决策全部交给模型。 

            ┌──────────────────────────┐
            ↓                          │
goal ──→ LLM ──plan──→ tool call ──→ env
            ↑                          │
            └────── observation ←──────┘
                  (until stop or done)

适用场景：开放式、探索性、步骤不可预知的任务，例如代码 Agent、调研 Agent 等。

4️⃣ 两者的结合

实践中两者经常混合使用：外层是 Workflow（保证主流程可控），内层关键决策用自主 Agent（处理开放子任务）。 这种 hybrid 模式能兼顾稳定性与灵活性，是大多数生产级 Agent 系统的真实形态。

五、Harness 展望

5.1 两条方法路径

  • Orchestration-first（编排优先）：通过更精细的流程设计来约束 Agent 行为，LangGraph 是代表
  • Harness-first（Harness 优先）：通过 AGENTS.md、Skills、结构化测试、运行环境等基础设施来引导 Agent 行为，Claude Code / OpenAI Codex 是代表

两条路径并非对立，而是互补——前者解决「流程的稳定性」，后者解决「长会话的自主性」。

5.2 Agent 架构的设计共识

  1. 简单优先：能单次调用解决的不上 Workflow，能 Workflow 解决的不上 Agent
  2. 可观测性：任何 Agent 决策都必须能被 trace、复现、回滚
  3. 权限最小化：默认禁止 destructive / external action，需显式批准
  4. 上下文可控：Prefix Cache、压缩、摘要等机制保证上下文不爆炸
  5. 错误纠正：让 Agent 在出错时能识别、汇报、修复，而不是崩溃

5.3 个人理解

Harness Engineering 不是取代 Prompt 或 Context Engineering，而是在更高层面回答同一个问题： 「怎么让 LLM 真正有用？」Prompt 解决单次调用质量，Context 解决单次会话质量，Harness 解决长期、自主、安全运行的质量。

对开发者来说，把「Agent 会犯错」这件事当作可工程化问题（而非随机事件）来对待，是 Harness Engineering 的核心心态—— 每一次错误都问：「是 prompt 不够好？是上下文不够？还是缺少工具/验证/权限设计？」 然后用工程手段让它下次不会再犯。
