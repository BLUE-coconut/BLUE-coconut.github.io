RAG（Retrieval-Augmented Generation，检索增强生成）是当下 LLM 应用最主流的工程范式之一。 本文梳理 RAG 的基础理论、文档预处理、向量存储与检索、构造 RAG 链以及测试优化等核心内容。

一、RAG 基本理论

RAG 定义

以检索增强生成为核心，将用户问题转化为检索请求，选出相关文档片段注入上下文，再由生成模型作答。

何时用 RAG

知识需要可溯源、常更新、规模大且无法完全「内化」到模型；希望减少幻觉并支持引用。

为什么引入 RAG

检索增强生成（RAG）通过语义相似性计算从外部知识库中检索相关文档块，从而增强了 LLM。 通过引用外部知识，RAG 可有效减少生成与事实不符内容的问题。

RAG 分类

根据技术实施阶段来分

分为 fine-tune、pre-training、inference 三种，即在模型的不同阶段引入 RAG 技术。

根据方法结构来分

  • 朴素 RAG：最基础的 RAG 实施方式。
    1. 对文档进行划分
    2. 对文档分模块进行特征编码（embedding）
    3. 根据用户输入对文档内容进行检索
    4. 将匹配的文档内容取出，与用户提示词一起，形成增强的 prompt 再输给 LLM
    5. LLM 生成答案
  • 高级 RAG：围绕检索前和检索后提出了多种优化策略，其过程与朴素 RAG 相似，仍然遵循链状结构。
  • 模块化 RAG：有更大灵活性，引入多个特定功能模块。

两大实现范式

  • Retrieval-then-Generate（最常见）：先检索再生成，把检索的 top-k 文本作为 context 传入 LLM，然后生成答案。
  • Generate-then-Retrieve / Iterative：先生成检索提示（如检索关键词或子问题），再检索并生成（用于复杂多步检索）。

RAG 基本模块

文档预处理与 chunking（切片）

对数据进行清洗，然后把大文档分成合理大小的片段。

  • 把大文档分成合理大小的片段（chunk），常见策略：
    • 固定 token/字符长度（例如 200-600 tokens）并重叠（overlap 50-100 tokens）
    • 基于语义边界（段落 / 小节 / 句子聚合）
  • 目标：每个 chunk 能保持语义完整且适配 LLM 的上下文窗口与检索精度
  • 注意：过短会丢失语义，过长会增加向量相似性噪音与 cost

Embeddings

将文本映射为向量。

  • 常用模型：sentence-transformers 系列（如 all-MiniLM-L6-v2）、OpenAI embedding（text-embedding-3-small）、Mistral/Anthropic embedding 等
  • 选择 embedding 时要考虑：语义质量、吞吐、GPU/CPU 适配、成本
  • 批量向量化时注意并发、batch-size、token 限制

Vector Store

向量存储方法。

  • 常见选项：FAISS（本地，高性能）、Chroma（易用）、Milvus（分布式）、Pinecone（托管）、Weaviate
  • 选择原则：数据量、延迟要求、是否需要持久化/并发、预算
  • Index 类型：flat / HNSW / IVF 等，通常 HNSW 适合低延迟高召回

Retriever

封装检索策略。

  • Dense Retriever（向量检索）：基于向量相似度（cosine / dot）匹配，MMR 最大边际相关性
  • Sparse Retriever（词袋/BM25）：基于关键字匹配，常用于初筛或混合策略
  • Hybrid Retriever：词袋 + 向量混合，兼顾精确与泛化
  • Cross-encoder Re-ranker（可选）：用 cross-encoder 模型对 top-n 再打分，提高精确度（代价大）

Context Injection

将检索的内容注入上下文，引导 LLM 生成回答。

  • Prompt 需约束「只用给定上下文」
  • 要确保注入的上下文 + query 不超模型上下文窗口。使用摘要或压缩上下文的方法来解决
SYSTEM: 你是雅思教师...
USER: {query}
CONTEXT: 【1】{doc1_summary} ... 【k】{dock_summary}
INSTRUCTIONS: 基于 above context 回答，并在答案末尾给出来源编号。

LLM Generator 的 tips

  • 直接生成（让 LLM 基于注入内容回答）
  • 验证/二次检索（LLM 回答后，再检索确认或补充）
  • 引用标注：使用「指令 + 强制引用」或「要求给出来源句子/段落」

常见挑战与解决策略

挑战策略
上下文太长 / Token 超限 对 chunks 摘要、优先保留高置信段落、动态截断；或检索-生成分步策略
信息冗余 vs 噪声 引入 re-ranker；或者把 chunk 中的关键句抽取后再注入
低召回 增大 chunk 多样性、使用 hybrid retriever、调节 embedding 模型
Hallucination（编造） 强制模型只基于 context 回答、要求「仅基于下列来源回答」，并在回答中附上来源句子或编号
延迟 本地 FAISS + HNSW，减少 re-ranker；缓存常见 query 的检索结果或生成结果
成本 减少 prompt 长度、减少 LLM token 使用、使用小模型在生成前做初筛/摘要

二、数据爬取

Step1：检查可抓取性（Robots 协议）

Robots 协议（Robots Exclusion Protocol）用来告诉爬虫和搜索引擎哪些页面可以抓取、哪些不可以。 一般放在网站根目录下的 robots.txt 中。 

import urllib.robotparser as rp

def can_fetch(base_url: str, path: str, ua="Mozilla/5.0 (Macintosh; Intel Mac OS X)"):
    robots = rp.RobotFileParser()
    robots.set_url(base_url.rstrip("/") + "/robots.txt")
    try:
        robots.read()
    except:
        return True
    return robots.can_fetch(ua, path)

Step2：建立 http 会话，抓取页面内容

import requests
from requests.adapters import HTTPAdapter
from urllib3.util.retry import Retry

def build_session():
    session = requests.Session()
    retry = Retry(total=4, backoff_factor=0.5,
                  status_forcelist=[429, 500, 502, 503, 504],
                  allowed_methods=["GET"])
    adapter = HTTPAdapter(max_retries=retry)
    session.mount("http://", adapter)
    session.mount("https://", adapter)
    return session

def fetch_page(url: str) -> str:
    session = build_session()
    headers = {"User-Agent": "Mozilla/5.0 (Macintosh; Intel Mac OS X) Safari/605.1.15"}
    resp = session.get(url, headers=headers, timeout=15)
    resp.raise_for_status()
    return resp.text

Step3：使用 BeautifulSoup 解析 HTML

recursive 的作用：recursive=True 表示递归向下遍历所有后代节点（默认）；recursive=False 则只在直接子节点里匹配。

from bs4 import BeautifulSoup

html = """
<div id="content">
  <ul class="outer">
    <li>Top-level 1</li>
    <li>Top-level 2
      <ul class="inner">
        <li>Nested A</li>
        <li>Nested B</li>
      </ul>
    </li>
  </ul>
</div>
"""

soup = BeautifulSoup(html, "lxml")
ul = soup.select_one("ul.outer")
print([li.get_text(strip=True) for li in ul.find_all("li", recursive=False)])
# ['Top-level 1', 'Top-level 2']
print([li.get_text(strip=True) for li in ul.find_all("li", recursive=True)])
# ['Top-level 1', 'Top-level 2', 'Nested A', 'Nested B']

常用入口与解析器

  • 构造 Soup：BeautifulSoup(html, "lxml") 更快更稳；html.parser 内置但性能稍弱
  • find / find_all：根据标签名或属性查找元素
  • select / select_one：根据 CSS 选择器查找（如 soup.select("div.question-item")）
  • 节点访问：tag.name、tag.attrs、tag.parent、tag.children、tag.get_text(separator=" ", strip=True)
  • 提取与清理：tag.decompose() 从树中删除；tag.extract() 摘除并返回

三、数据预处理

数据清洗

  • 统一格式（去除无关序号、空格、特殊符号）
  • 去除垃圾信息（网页中的广告、导航信息等）
  • 可选：信息归类

数据 Chunk

  • 对于独立性高的短文本（如题目）一个片段对应一段文本
  • 段落/句子切分：对较长 sample_answer 按段落或句子分割
  • 字符长度切分：固定 chunk_size 与 overlap（如 600/80）
  • Token 长度切分：用近似 token 计数控制上下文长度
  • 语义切分（可选）：按语义边界拆分或聚类
  • 结构切分：按 Part/Topic 建层级或聚合

四、数据向量化、向量存储与检索

第 1 步：选型

  • Embedding 选项：all-MiniLM-L6-v2（小、快、本地）、OpenAI text-embedding-3-small（语义质量高但有成本）
  • Vector DB 选项：FAISS（本地）、Chroma（易用）、Milvus/Weaviate（分布式）、Pinecone（托管）
  • 开发建议：MiniLM + FAISS，或 OpenAI + Chroma

第 2 步：文本向量化 Embedding

from sentence_transformers import SentenceTransformer
model = SentenceTransformer("all-MiniLM-L6-v2")

def chunk_batches(chunks, batch_size=32):
    for i in range(0, len(chunks), batch_size):
        yield chunks[i:i+batch_size]

for batch in chunk_batches(chunks, batch_size=64):
    texts = [c["text"] for c in batch]
    embeddings = model.encode(texts, show_progress_bar=False, convert_to_numpy=True)

第 7 步：数据检索方法

相似度检索

from langchain_chroma import Chroma
from langchain.embeddings import OpenAIEmbeddings

emb = OpenAIEmbeddings()
vectordb = Chroma(persist_directory="chroma_store", embedding_function=emb)

docs = vectordb.similarity_search("how to use moreover in essay?", k=5)
docs_with_scores = vectordb.similarity_search_with_score("how to use moreover?", k=10)

MMR（Maximum Marginal Relevance）

retriever = vectordb.as_retriever(
    search_type="mmr",
    search_kwargs={"k": 5, "fetch_k": 50, "lambda_mult": 0.5}
)
docs = retriever.get_relevant_documents("compare affect vs effect")
  • fetch_k：先从向量库取多少候选
  • lambda_mult：多样性系数（0→只按相似度；1→更注重多样性）

六、构造 RAG 链

retriever = vectordb.as_retriever(search_type="similarity", search_kwargs={"k":4})
from langchain.chains import RetrievalQA
qa_chain = RetrievalQA.from_chain_type(llm=chat_model, chain_type="stuff", retriever=retriever)

七、测试优化方法

质量检测与监控建议

  • Sanity checks：检索返回的 top1 distance 是否在合理范围
  • Sample verification：抽样 5% 检索结果，人工标注相关性
  • Logging：记录 query、retrieved_ids、retrieved_scores、response_time
  • Alerting：当平均 top1 distance > 阈值 或 检索失败率上升时报警
  • Caching：对高频 query 缓存 top-k 结果（如 Redis）

性能与成本优化

  • 减少 embedding 成本：先用小模型做 candidate filtering
  • 降低延迟：本地 FAISS + HNSW；并发查询（注意锁与线程安全）
  • 压缩索引：使用 PQ/IVF 进行量化
  • 批量处理：embedding 批量化，入库批量化
  • GPU：FAISS-GPU 或 Milvus GPU 实例

小结

RAG 的核心价值是把检索能力嵌入生成链路——让 LLM 在生成时获得准确、可追溯、可更新的外部知识。 工程上的关键点不在某一个具体环节，而在于各环节之间的协同：chunking 决定 Embedding 语义质量， Embedding 与 Vector Store 决定召回率，Retriever 策略决定 top-k 相关性，Context Injection 决定 LLM 能否合理利用检索结果。

高质量数据 + 合理切分 + 可解释的检索 + 严格的生成约束 + 持续评估，是 RAG 系统从 demo 走向生产的关键。
