Claude Code 的能力组合由多种机制叠加而成：内置 Tool 是执行底座， MCP 与 CLI 分别以「结构化协议」和「shell 调用」两种方式接入外部能力， Skill 为 Agent 注入程序性知识以编排上述能力，Claude Code Plugin 把多种组件打包以便分发与共享。

一、MCP —— 连接外部工具

什么是 MCP

MCP（Model Context Protocol）是一个开放标准协议，定义了 AI 应用与外部工具/数据源之间的通信方式。 它一般由第三方服务平台（例如 GitHub、数据库、Slack、飞书 等）提供， 以方便任意外部系统的能力能够快速接入 Agent 系统中被 AI 调用。

基本架构：Host → Client → Server

MCP 采用三层架构：

  • Host：AI 应用（如 Claude Code），负责协调多个 Client
  • Client：Host 内部组件，每个 Client 维护与一个 Server 的专属连接
  • Server：独立进程，向 Client 暴露工具（Tools）、资源（Resources）、提示词模板（Prompts）

所有通信基于 JSON-RPC 2.0 协议，与传输方式无关。

MCP 的调用机制

Server 端：用 JSON Schema 声明工具

Server 在响应 tools/list 时返回工具列表，每项包含 name、description、inputSchema（标准 JSON Schema）。 其中，description 是模型决定何时调用该工具的唯一线索——写得越具体准确，误调用与漏调用越少。

Agent 端：加载与路由工具

Claude Code 启动后的工具加载流程：

  1. 读取 .mcp.json / settings.json 中的 Server 配置
  2. 为每个 Server 建立连接（STDIO 拉起子进程 / HTTP 建立 SSE 长连）
  3. 发送 initialize 协商协议版本与 capabilities
  4. 调用 tools/list 拉取所有工具的完整 schema
  5. 按 mcp__<server名>__<tool名> 重命名每个工具（避免跨 Server 冲突）
  6. 将转换后的工具列表注入到每次发往 Claude API 的 tools 字段中

特点：启动时全量加载，运行时全量注入。

模型调用工具的流程

返回 tool_use block → Claude Code 按前缀路由到对应 Server 发送 tools/call → Server 返回 content → Claude Code 包装成 tool_result 注入下一轮对话。 MCP 协议只存在于 Claude Code 与 Server 之间——这种「协议在 Host 内消化」的设计，是 MCP 能成为跨厂标准的关键。

代价是 token 开销：所有工具的 schema 必须常驻在每次请求的 tools 字段中， 否则模型就「看不到」这些工具——这与下一节 CLI 的按需发现形成对比。

两种连接方式

核心区别在于：谁来管理 Server 的生命周期。

STDIO — 本地子进程模式

Claude Code 在本地拉起 Server 作为子进程，通过 stdin/stdout 通信。Server 的整个生命周期由 Claude Code 管理——会话开始时启动，会话结束时终止。 Server 程序可以预先安装在本地，也可以通过 npx、uvx 等工具临时下载运行（无需持久安装）。

  • 一个 Server 实例只服务一个 Client
  • 无网络开销，性能最优
  • 典型场景：日常使用的大多数 MCP Server
claude mcp add github -- npx -y @modelcontextprotocol/server-github

Streamable HTTP — 独立服务模式

Server 是一个独立运行的 HTTP 服务，不由 Claude Code 管理生命周期。Claude Code 作为 HTTP 客户端去连接它。服务可以运行在本地、内网或云端。

  • 一个 Server 可同时服务多个 Client
  • 支持标准 HTTP 认证和会话管理
  • 典型场景：企业级部署、多用户共享、第三方平台集成
claude mcp add my-server --transport sse --url http://localhost:3001/sse

MCP 的价值

MCP 解决了「能力边界」问题——通过开放协议把外部系统的能力转译为 Agent 可调用的工具， 让任何模型供应商都能接入同一个生态。但 MCP 只提供工具，不提供使用方法——「怎么用好这些工具」是 Skill 的职责。

二、CLI —— 命令行接口

什么是 CLI

CLI 即服务方提供的命令行可执行接口，例如 gh（GitHub 命令行工具）、lark-cli（飞书命令行工具）等。 在 Agent 语境下，CLI 是另一种把「外部能力」暴露给 LLM 的工程范式：通过 shell 而非 RPC 协议。

Agent 如何调用 CLI

Agent 调用 CLI 不引入任何新协议——所有交互都借由 Agent 内置的 Bash Tool 完成：

  1. 执行：模型在响应中返回一次 Bash 工具调用，命令字符串作为参数
  2. 捕获：Bash Tool 在子进程中执行该命令，把 stdout / stderr 文本与 exit code 一并回传给 Agent
  3. 判断：Agent 把这些文本作为 tool_result 注入下一轮上下文，依据 exit code 决定继续推理或重试

整个过程对模型而言与调用任意其它工具完全一致——它只需要会写正确的 shell 命令。

关键机制：按需发现（lazy discovery）

这是 CLI 与 MCP 在工具发现机制上的根本分歧。 MCP 客户端在启动时通过 tools/list 把所有工具的完整 schema 一次性注入系统提示； CLI 则是让 Agent 通过 --help 在执行过程中递归探索命令树： 

$ gh --help              # Agent 先看顶级命令分类
$ gh pr --help           # 选定 pr 域后看 pr 的子命令
$ gh pr create --help    # 选定 create 后看具体参数

只有 Agent 当前任务真正需要的子命令的 --help 输出才会进入上下文。例如，在一个仅需创建 issue 的任务中，PR、release、actions 等命令的 schema 不会全部一次性传给 Agent。

这是 CLI 在 token 经济性上根本优势的来源。

CLI 与 MCP 的区别

CLI 与 MCP 在「为 Agent 提供外部能力」这一职能上是同位的，但走了不同的工程路径。最核心的差异点在于：「工具的描述」什么时候被 LLM 看到？

  • MCP：会话开始时全量注入。客户端启动时调用 tools/list 拉取所有工具完整 schema，注入每次请求的 tools 字段。未在启动时加载的工具，模型完全感知不到。
  • CLI：任务真正需要时按需发现。Agent 通过 --help 在 Bash 调用中递归探索命令树。

CLI 的价值

CLI 不是对 MCP 完全的替代——而是「如何让 Agent 调用外部能力」的两种工程答案： MCP 通过协议把外部系统的能力结构化暴露给 LLM；而 CLI 让 LLM 用最熟悉的 shell 调用第三方服务的命令行接口。 前者是为 Agent 重新设计的外部系统接口标准，后者是依托 shell 这一同时服务人类与 Agent 的通用接口范式。

三、Skill —— 注入领域知识

什么是 Skill

Skill 是一个装载领域知识与可执行资源的目录——它不是单个 Markdown 文件， 而是包含 SKILL.md 元数据 + 可选的脚本、参考文档、模板资源的包。 Anthropic 工程团队对 Skill 的官方定义：

"A skill is a directory containing a SKILL.md file that contains organized folders of instructions, scripts, and resources that give agents additional capabilities."

该格式由 Anthropic 在 2025 年 10 月推出，2025 年 12 月作为开放标准发布于 agentskills.io。 已被 30+ 个 Agent 框架 原生支持，包括 Claude Code、OpenAI Codex CLI、Google Gemini CLI、GitHub Copilot、Cursor 等——是继 MCP 之后又一个被业界广泛接纳的 Agent 扩展标准。

⭐️ 核心机制：渐进式披露

Skill 的设计目标是解决一个根本矛盾：Agent 需要尽可能多的领域知识，但又不能让所有知识常驻在上下文中。

核心方法是 Progressive Disclosure（渐进式披露）——把信息分成三层，按需逐级加载：

层级API 调用加载内容Token 消耗
Level 0skills_list()所有技能的 name + description约 3K（固定）
Level 1skill_view(name)完整 SKILL.md按需
Level 2skill_view(name, path)特定 reference/script 文件按需

其结果是：Skill 中可承载的知识量在工程上是无界的—— 只要分层组织得当，Agent 总能在需要时找到需要的信息。

Agent Skills 标准结构

一个技能就是一个文件夹，最简单的只需要一个 SKILL.md 文件：

skill-name/
├── SKILL.md              # 必需：元数据 + 指令
├── scripts/              # 可选：可执行脚本
├── references/           # 可选：参考文档
└── assets/               # 可选：模板、资源文件

硬性约束：SKILL.md 文件名大小写敏感；frontmatter 中的 name 字段必须严格等于所在目录名。

标准 frontmatter 仅有 6 个字段，其中 2 个必需：

字段必需说明
name✓技能名；仅小写字母 / 数字 / 连字符；必须等于目录名
description✓必须同时说明「做什么」和「何时使用」——是 Agent 决定调用技能的关键信息
license✗协议名或 license 文件引用，如 Apache-2.0
compatibility✗环境要求（适配产品、系统包、网络访问等）
metadata✗通用扩展点；version、author 等放此处
allowed-tools✗预批准的工具，如 Bash(git:*) Read

Skill 的价值

Simon Willison 的判断是 "Claude Skills are awesome, maybe a bigger deal than MCP"—— 他给出的核心理由是：Skill 把「如何做事」的工程实践外化为标准化、可移植的工件， 而 MCP 只解决了「能调用什么」。

四、Plugin —— 打包分发能力集合

什么是 Plugin

Plugin 是一个 Claude code 中的「能力合集的安装包」，把 Skills、Hooks、MCP 配置、自定义 Agent 等多种组件打包成一个可安装的包。

组件目录作用
Skillsskills/领域知识和操作流程
Agentsagents/自定义 Subagent 人格/工具集
Hookshooks/生命周期自动化（格式化、安全检查）
MCP.mcp.json外部工具连接配置
可执行文件bin/加入 PATH 的原生工具

它解决的是「如何把一套能力共享给团队或社区」的问题。安装一个 Plugin，就同时获得了它包含的所有 Skill、Hook、MCP Server 等。

目录结构

my-plugin/
├── .claude-plugin/
│   └── plugin.json        # 清单文件（必需）
├── skills/                # Skill 集合
│   ├── code-review/
│   │   └── SKILL.md
│   └── security-audit/
│       └── SKILL.md
├── agents/                # 自定义 Subagent
│   └── reviewer.md
├── hooks/                 # Hook 配置
│   └── hooks.json
├── .mcp.json              # MCP Server 配置
├── bin/                   # 可执行工具（加入 PATH）
│   └── my-tool
└── README.md

清单文件（plugin.json）

{
  "name": "team-standards",
  "description": "团队开发规范：代码审查、安全检查、提交规范",
  "version": "1.0.0",
  "author": { "name": "DevTeam" },
  "tags": ["code-review", "security"]
}

Plugin 的本质

Plugin 解决的是「分发」问题——它不引入新概念， 只是把已有的 Skill、Hook、MCP 等组件打包在一起，加上版本管理和命名空间隔离，使其可以在团队和社区间共享。

层次关系

五者构成由「原子能力」到「分发单元」的递进层级，互补协作而非彼此替代：

  • Tool 是 Agent 的基础执行单位（Bash、Read、Edit），构成所有外部交互最终落地的「硬件层」
  • MCP 通过开放协议把外部系统封装为结构化工具，适合需要类型安全与协议级特性的场景
  • CLI 通过 Bash Tool 调用既有的命令行程序，是当前 Token 经济性最优、生态覆盖最广的接外部能力路径
  • Skill 以程序性知识告诉 Agent 应当如何编排上述 Tool / MCP / CLI，沉淀为可复用工序
  • Plugin 把若干 Skill、Hook、MCP 配置打包并加上版本与命名空间，使能力可分发到团队与社区
