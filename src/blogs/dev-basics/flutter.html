一、Widget 基础

Flutter 中的四棵树

  1. Widget：描述 UI 的配置信息
  2. Element：根据 Widget 的描述生成真正的实例
  3. Render：应用界面的布局和绘制，保存了元素的大小、布局、颜色等信息
  4. Layer

Widget 分类

在 Flutter 中，widget 用来描述一个 UI 的配置信息（包括内容、布局信息等），是不可变的（immutable）。

Widget 有两类：

  • StatelessWidget：无状态
  • StatefulWidget：有状态。由于 widget 是不可变的，对于有状态的组件，需要引入 state 来描述其布局和内容。可以跟随状态（即变量）进行变化，然后根据该 state 生成 widget，再进行页面更新。

当 widget 发生改变时，会触发框架重建 widget 树。Flutter 会遍历 widget 树中每个节点，逐一比较新的 widget 和原先的 widget：

  • 如果新老 widget 为同一类型，但配置改变，仅修改对应 RenderObject 的配置
  • 如果新老 widget 不为同一类型，则将对应的 widget、element、RenderObject 节点从树上移除，并用新的对象替换

二、状态管理

状态管理方式

  1. 组件管理自己的状态：一个组件的颜色、动画等外观属性最好采取这种方式
  2. 由父组件管理状态：全局的用户信息（例如：表单中复选框的选中状态、滑块的位置）& 有些状态需要几个组件共享使用的情况
  3. 混合管理：既有自己管理的部分，也有父组件管理的部分

状态管理实现

  1. 管理自身状态：使用 setState() 更新 UI 状态
  2. 管理子组件状态：用回调函数。将包含状态更新（即 setState）功能的函数传递给子组件，让子组件在合适的地方调用。

关于回调函数

除了使用普通的 Function(传参类型) 外，Flutter 中有以下预定义的回调函数类型：

三、路由管理

使用路由跳转页面

  1. Navigator 是路由管理组件，提供打开和退出路由的方法：
    • Navigator.push(context, route)：路由入栈，打开一个新页面，返回值是一个 Future 对象
    • Navigator.pop(context, [result])：路由出栈，关闭页面
  2. MaterialPageRoute 继承自 PageRoute，定义了路由构建及切换时过渡动画的相关接口。能够针对不同平台实现与平台页面切换动画风格一致的过渡：Android 是上下滑动，iOS 是左右滑动。

示例

class TipRoute extends StatelessWidget {
  TipRoute({Key key, required this.text}) : super(key: key);
  final String text;

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      appBar: AppBar(title: Text("提示")),
      body: Padding(
        padding: EdgeInsets.all(18),
        child: Center(
          child: Column(
            children: <Widget>[
              Text(text),
              ElevatedButton(
                onPressed: () => Navigator.pop(context, "我是返回值"),
                child: Text("返回"),
              )
            ],
          ),
        ),
      ),
    );
  }
}

命名路由

注册路由表：

MaterialApp(
  title: 'Flutter Demo',
  initialRoute: "/",
  theme: ThemeData(primarySwatch: Colors.blue),
  routes: {
    "new_page": (context) => NewRoute(),
    "/": (context) => MyHomePage(title: 'Flutter Demo Home Page'),
  }
);

用路由名打开页面：

onPressed: () {
  Navigator.pushNamed(context, "new_page");
},

对于命名路由，通过 arguments 传参：

Navigator.of(context).pushNamed("new_page", arguments: "hi");

路由跳转前置处理

典型适用场景：访问需要登录的页面，在 onGenerateRoute 中判断并拦截：

MaterialApp(
  initialRoute: "index",
  onGenerateRoute: myGenerateRoute,
);

String routeBeforeHook(RouteSettings settings) {
  final token = Global.prefs.getString('token') ?? '';
  if (token != '') {
    if (settings.name == 'login') return 'index';
    return settings.name;
  }
  return 'login';
}

Route<dynamic> myGenerateRoute(RouteSettings settings) {
  String routeName = routeBeforeHook(settings);
  return MaterialPageRoute(builder: (context) {
    switch (routeName) {
      case "index": return MyHomePage();
      case "login": return LoginScreen();
      default: return Scaffold(body: Center(child: Text("页面不存在")));
    }
  });
}

四、Key 的原理和用法

Key 的作用

每个 widget 都有一个 Key，能够解决类似「不同 widget 混淆」的问题。

Key 的种类

  • LocalKey（局部键）：在同一级中（对于 widget）要唯一，可以理解为「同级唯一性」。
    • ValueKey、ObjectKey、UniqueKey
  • GlobalKey（全局键）：在整个 App 中必须唯一。即使 widget 树的节点改变，其内容（GlobalKey 定义的变量）不改变。

使用 GlobalKey 不应该在每次 build 的时候重建 GlobalKey，它应该是 State 拥有的长期存在的对象。

获取对应的 State

// 获取当前组件的 count 变量
floatingActionButton: FloatingActionButton(
  onPressed: () {
    print((_globalKey.currentState as _BoxState).count);
  },
  child: Icon(Icons.wifi_protected_setup),
)

// 修改组件的 count 变量
onPressed: () {
  final state = _globalKey.currentState as _BoxState;
  state.setState(() {
    state.count++;
  });
}

注意：修改 state 时，记得使用 setState 来触发 UI 的刷新。

获取对应的 context（用于获取 widget 的尺寸、位置）

final renderBox = _globalKey.currentContext!.findRenderObject() as RenderBox;
renderBox.size;                        // 获取尺寸
renderBox.localToGlobal(Offset.zero);  // 获取位置

获取对应的 widget（如颜色等属性）

onPressed: () {
  final widget = _globalKey.currentWidget as Box;
  print(widget.color);
}

五、Provider 库的使用

Provider 库用于「跨页面、跨组件共享 + 会变化」的场景。

典型场景

  1. 全局状态 / 会话级
    • 用户登录态（AuthProvider）
    • 主题 / 国际化（ThemeProvider、LocaleProvider）
    • 环境配置（FlavorProvider、FeatureFlagProvider）
  2. 页面级共享
    • 购物车（CartProvider）
    • 筛选条件（FilterProvider）
    • 分页列表控制器（PagingProvider）
  3. 复杂表单 / 多步向导
    • 表单数据模型（FormProvider）
    • 步骤状态机（WizardProvider）
  4. 异步一次性数据
    • 网络请求结果（FutureProvider）
    • WebSocket 消息流（StreamProvider）
  5. 需要 dispose 的资源
    • 蓝牙连接（BleClient）
    • 数据库 DAO（AppDatabase）
    • 计时器 / 动画控制器

三种使用方法

// 1. Consumer —— 监听状态变化，自动重建
Consumer<VocabularyProvider>(
  builder: (context, provider, child) {
    return Text('单词数量: ${provider.words.length}');
  },
)

// 2. context.watch —— 简洁的监听方式
Widget build(BuildContext context) {
  final provider = context.watch<VocabularyProvider>();
  return Text('单词数量: ${provider.words.length}');
}

// 3. context.read —— 不监听，只获取实例（用于事件处理）
void _addWord() {
  final provider = context.read<VocabularyProvider>();
  provider.addWord(newWord); // 不会重建当前 Widget
}

小结

Flutter 的核心心智模型可以总结为「声明式 UI + 不可变 Widget + 状态驱动重建」。 理解四棵树（Widget / Element / Render / Layer）以及它们之间的对应关系，是理解 Flutter 性能与渲染管线的基础； 状态管理的关键不在于选择哪种库（setState、Provider、Riverpod 等），而在于「状态属于谁、由谁负责更新」的设计划分。
