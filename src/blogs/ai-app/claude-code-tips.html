Claude Code 不只是「写代码的 ChatGPT」——它内置了一套成熟的会话管理、上下文控制、可观测性能力。 本文整理 6 个让日常体验明显提升的高阶技巧：撤销、重命名、洞察、状态栏、压缩、配额。

一、撤销对话与代码（/rewind）

写到一半发现「刚才那个改动方向错了」——以前只能 git checkout， 现在 Claude Code 自带会话级时光机。

触发方式（任选其一）

  • 连按两次 Esc（最快）
  • 输入 /rewind（别名 /checkpoint、/undo）

打开一个可滚动菜单，列出会话中你的每一条消息作为 checkpoint，选中后有四种回退方式：

选项行为
Restore code and conversation对话和代码都回到该点
Restore conversation只回退对话，代码保留
Restore code只回退代码，对话保留
Summarize from here从该点起，把后续消息压缩成摘要（保留上文完整）

重要陷阱

  • 只追踪 Claude 用编辑工具改的文件。Bash 命令改动的文件（rm、mv、脚本生成）不会被追踪——这种场景仍要靠 git
  • Checkpoint 默认 30 天后自动清理（可在 settings 里改）
  • Checkpoint ≠ Git。它是会话内的本地撤销，不是版本控制；离开会话后，git 才是唯一靠谱的历史

二、会话重命名（/rename）

默认 session 在 /resume 里只有 UUID 和首条消息预览，多了之后完全分不清谁是谁。 /rename 命令能够自行定义会话名称，可以将一些重要会话进行标识，方便后续 /resume 时快速找到。

用法

# 会话中
/rename                     # 不带参数：基于内容自动生成
/rename feishu-doc-refactor # 显式指定名字

# 启动时
claude -n "feishu-doc-refactor"
claude --name "feishu-doc-refactor"

显示位置

  • prompt bar 顶端
  • /resume 选择器列表（在选择器中按 R 也能快速重命名）
  • statusline JSON 字段 session_name（可被自定义状态栏脚本读取）

小技巧：重要的会话记得 /rename，方便后续定位找到要继续的对话。

三、会话洞察（/insights）

这是个相对冷门但很有意思的功能：让 Claude 分析你最近的使用习惯。

触发

/insights

做什么

扫描本地最近 30 天的会话历史，生成一份交互式 HTML 报告， 会放在 ~/.claude/usage-data/report.html（若没有自动跳转，可手动打开），自动在浏览器打开。 报告内容大致有：

  • 你最常处理的项目区域
  • 高频交互模式
  • 「摩擦点」——比如某类指令反复需要 3-5 轮纠正才能达到目的
  • 建议：把重复出现的指令沉淀进 CLAUDE.md

纯本地处理，不上传任何数据。每月/每季度回顾一次，能发现自己没意识到的习惯，进而沉淀成 CLAUDE.md 或 skills。

四、自定义状态栏（/statusline）

可自定义终端对话输入框下方的状态栏所陈列的信息，例如 token 用量、当前模型、上下文用量等。

最快配置：自然语言让 Claude Code 自行生成

/statusline show model name, context usage percentage, git branch, and total cost

Claude 会自动写一个脚本并更新 ~/.claude/settings.json。

手动配置

在 ~/.claude/settings.json 里加：

{
  "statusLine": {
    "type": "command",
    "command": "~/.claude/statusline.sh",
    "padding": 2,
    "refreshInterval": 5
  }
}

脚本能拿到的关键 JSON 字段

字段含义
model.display_name当前模型名
cwd / workspace.project_dir当前目录
cost.total_cost_usd累计花费
context_window.used_percentage上下文已用百分比（看是否该 /compact 的关键指标）
rate_limits.five_hour.used_percentage5 小时配额使用率
session_id / session_name会话 ID 和你 /rename 设置的名字

状态栏脚本会在每条新消息后跑一次，避免里面调慢命令。git 之类的输出建议加缓存。

五、上下文压缩（/compact）

/compact 和 /clear 经常被混用，但行为完全不同。

命令对话历史代码改动适用场景
/clear完全清空重开不动任务彻底切换、和当前没有上下文关联
/compact [指令]压缩成摘要保留要点不动同一任务继续做，但上下文已经吃紧

手动用法

/compact 重点保留架构决策和未完成的 TODO，可以丢弃中间调试细节

带参数会指导摘要的侧重点，不带参数走默认策略。

上下文用到约 83.5% 时（剩约 33K tokens 缓冲区）会自动 compact。

最佳实践

  • 别等自动触发——质量会明显下滑。养成在 ~60% 主动 /compact 的习惯（看状态栏百分比）
  • 关键规则放在 CLAUDE.md，不要只口头告诉 Claude——/compact 后口头指令可能丢失，但 CLAUDE.md 永远在
  • 单文件过大时频繁 compact 会引发 thrashing，这种情况下 /clear 更干脆

六、查看用量与配额（/stats）

/stats 查看 Claude Code 近期活动统计：

  • 所用对话累计成本（API 计费用户看 USD 数字；订阅用户看不到具体金额）
  • 5 小时配额已用百分比 + 下次重置时间
  • 7 天配额已用百分比（订阅用户最重要的指标）
  • 活动统计：消息数、各模型的 token 分布、工具调用次数

输入 r 可切换统计的时间段

小结

Claude Code 的这些「隐形功能」反映出一个核心设计哲学：把可观测性和可回滚性做进工具本身。 习惯使用 /rewind、/rename、/statusline、/compact 这四条之后， 日常开发的「摩擦成本」会明显降低——尤其是复杂长任务场景。

最后，把重复指令沉淀进 CLAUDE.md、每季度跑一次 /insights 复盘自己的工作流， 是真正能让 Claude Code 从「能用的工具」变成「值得托付的工具」的两个关键习惯。
